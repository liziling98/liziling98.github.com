项目

新闻文本分类
问题描述：80万新闻，均属于长文本，一共752个标签，大体分成工业，区域和话题三个大类，每条新闻对应若干条标签。现在要预测出每条新闻上可能附带的所有变迁，属于多分类问题。

标签主要使用独热编码，将具体的标签转化成0和1组成的矩阵。

传统机器学习中，作为预处理，分词后去除了停用词和标点符号，然后提取了词干，并且对词性进行还原。为了把单词转化成数值，经过预处理后，计算了他们的TFIDF值；

在深度学习方法中，清洗文本后，先将所有单词转化成索引，再用斯坦福官方发布的glove将词索引转化成词向量，在训练中不对词向量进行更新。

在这个任务中，有多少个标签，就训练了多少个逻辑回归，这里采用了问题转化方法，把多分类划分多个二分类的任务。

决策树使用了算法适应的方法，直接训练一个决策树分类器来预测所有的标签，它能直接处理多标签数据。

CNN使用了双层卷积，以及5个核大小不一样的filter来同时对文本进行卷积，卷积结束后将所有结果联列在一起，卷积和激活函数之间使用batchnorm取代dropout来避免过拟合，全连接层使用了两层linear，其中也是用了batchnorm来避免过拟合。CNN在新闻分类task上表现不错的原因是，虽然它忽视了上下文关系，但它能根据关键词来做出判断，这很符合人类对新闻分类时的操作，根据关键词判断新闻属于什么类别。

用fasttext来作为baseline，因为它主要使用线性矩阵运算，在gpu上运行收敛速度快，效果也不错。fasttext在两个全连接层之间计算平均值。

评估同时考量了宏平均和微平均，考量宏是为了观察分类器在所有标签类别上的综合表现，不管标签出现是否频繁，而考量微是为了观察分类器整体的表现，它不太受稀有标签影响。

保险索赔预测
问题提出：使用车辆特征预测保险索赔金额

主要挑战：

  1. 索赔和变量之间关联性不强
  2. 数据维度高，30多维
  3. 缺失数据且数据量大

预处理：数据集是缺少数据，要处理里面的空值，删除或者插补均可；有的数据是类别数据，将其转化成独热编码。数据高度不平衡，大多数记录包含零索赔。为数据贴标签，存在索赔的设置为1，索赔金额为0的设置成1

特征：

  1. 生产年份
  2. 制造商
  3. 引擎类别数据，共12列，每列下不同的类别比如ABC，不同列下ABC互不相等
  4. 引擎数值型数据，8列，均值为0，标准差为1
  5. 非引擎类别数据，ABC等
  6. 非引擎数值型数据，4列，均值为0，标准差为1
  7. 索赔金额，99%是0

数值型数据中，引擎数值型数据和非引擎数值型数据变量之间的相关系数为负，和索赔金额相关系数为负，非引擎数值型数据和索赔金额相关系数为正

首先，移除了4个引擎类别数据，因为在这4列下，超过50%的值是空值，删除后，如果空值仍然出现，直接删除列，1300万条数据共删除7万条。将类别数据转化成独热编码后，联列在一起，使用PCA降维，因为维度太高，计算速度慢。oversample非零索赔值得数据，然后在分类时，调整相应的权重，对于非零值，使用0.99，零值使用0.01.

其次，维度太高，需要降维，使用PCA，处理12列数值型数据，然后处理转换成one hot encoding类别的类别型变量

先用逻辑回归判断是否存在索赔额，再对预测为正类的条目用广义线性回归（family = ‘gamma’, link = ‘log’）算法计算索赔额，最后的AUC值为0.54

Pyspark中主要使用的函数包括select, filter, withColumn, withColumnRenamed, groupBy, union,

系统失败检测

检测，换而言之，追踪

Wandisco：分布式数据平台，WANdisco平台的核心目标是确保数据可在不同的区域之间复制，并且在每个区域中访问时都是一致的。

问题提出：WANdisco Fusion的分布式架构系统日志量很大，来自不同节点的日志中具有重复和关联性，利用重复和相关的结构来追溯事件的失败。直接从命令行可以看到log文件，但很难根据肉眼看出这些日志文件数据的趋势和它们运行失败的模式，这就很难直接了解到导致系统出错的原因。

项目步骤：

  1. 对观察到的日志进行分类和标记，寻找到有关各个事件成功和失败信息相关的记录和字段，以及提供有关系统整体运行状况信息的字段，以创建数据模式。
  2. 使用这些数据提供单个事件故障和/或节点故障的示例，识别导致失败的特定事件结果或系统特征之间的趋势，模式或相关性，并且描述观察到的相关性或趋势之间的任何关系或因果关系

一共有4个节点的数据，其中有一个节点存在错误。几种主要的log：

  1. 服务器日志：包括时间戳，日志的错误等级（恐慌，严重，错误，警告，普通INFO，PANIC将始终导致Fusion服务器关闭，SEVERE可能会导致系统无法正常运行。错误标志系统中出现意外问题，警告有时会突出显示正常故障），用户，和日志文本信息，从日志文本信息中可以提取出当前状态，请求ID，节点ID，唯一标识ID，和请求状态
  2. JAVA垃圾回收日志：garbage first垃圾回收机制，主要记录了系统运行时的堆变化（内存变化）
  3. 系统运行日志：包括 %commit and %iowait等

特征工程：

  1. 错误率：按一个固定的时间段，计算错误信息和总信息条数的比率，比如恐慌信息占比，警告信息占比等等
  2. iowait：变动越大，失败（表示在一个采样周期内有百分之几的时间属于以下情况：CPU空闲、并且有仍未完成的I/O请求）
  3. commit ratio：上下浮动大，失败（内存申请占物理内存与交换区之和的百分比）
  4. 垃圾回收堆变化率：垃圾回收后堆内存变化率

发现失败节点的错误率相比正常节点的错误率变化更大，错误率更高。

根据时间戳，将各个特征作为多维时间序列数据集。使用kmeans算法聚类
